## プロジェクト概要と統合アーキテクチャ

### 1.1 プロジェクトの核心

本プロジェクト（仮称：Proteus）は、古代ギリシャ語をはじめとする古代地中海諸語の碑文・パピルス・文学テキストにおける音韻変異（方言差、時代差、表記ゆれ）を体系的に検索・発見するためのフレームワークである。
既存のデジタル古典学ツール（TLG、Perseus、PHI、papyri.info）は全て正字法ベースの検索を前提としており、音韻パターンによる検索、方言変異の体系的解決、適用された音変化ルールの説明を提供するツールは一つも存在しない。この構造的ギャップを埋めることが本プロジェクトの目的である。

### 1.2 SCAMPER分析から導出された5つの柱

| 柱 | 名称 | 概要 |
|---|---|---|
| ① | コアアルゴリズム | バイオインフォマティクス由来の音韻距離行列 + BLAST的3段階検索 |
| ② | 多言語フレームワーク | 言語非依存設計。ギリシャ語→ラテン語→コプト語と拡張可能 |
| ③ | MCPサーバー | LLM向けツール提供。Claude/GPTが音韻検索を参照して回答 |
| ④ | アノテーションエンジン | コーパス全体を事前スキャンし、音韻変異を自動検出・フラグ付け |
| ⑤ | フェデレーテッド検索 | コーパスを保有せずインデックスのみ保持。既存DBへのリンクで参照 |

### 1.3 統合アーキテクチャ概念図

本フレームワークは以下の5層で構成される。各層は独立して開発・テスト可能であり、Phase 0〜3で段階的に構築する。
- 【ユーザー層】 研究者（Web UI）/ DHプロジェクト（REST API）/ LLMエージェント（MCPサーバー）/ 一般ユーザー（LLM経由）
- 【インターフェース層】 Web UI + REST API + MCPサーバー + Linked Open Data（SPARQL）
- 【実行モード層】 Pull型（検索エンジン）+ Push型（アノテーションエンジン）
- 【フレームワーク層】 言語非依存の検索アルゴリズム + インデックス構造 + 説明生成エンジン
- 【言語プラグイン層】 各言語の音素インベントリ / 表記⇔音素マッピング / 音韻距離行列 / 音韻変化ルールセット

### 1.4 技術的前提条件

本ロードマップは、開発者がAI支援コーディングツール（Claude Code、Codex等）を主な実装手段として使用することを前提に設計されている。各Phaseの実装仕様は、これらのツールに「プロンプト」として渡せる粒度で記述する。

主要技術スタック（推奨）：
- 言語：Python 3.11+（学術NLPエコシステムとの親和性が最も高い）
- 音韻処理：LingPy（比較言語学ライブラリ、pip install lingpy）
- Web API：FastAPI（Python、非同期対応、自動ドキュメント生成）
- フロントエンド：シンプルなHTML + JavaScript（Phase 0）→ React（Phase 1以降）
- データ格納：SQLite（Phase 0）→ PostgreSQL（Phase 1以降）
- デプロイ：Render / Vercel / Railway（無料枠あり）
- MCPサーバー：Python MCP SDK（pip install mcp）
- バージョン管理：GitHub（公開リポジトリ）
- CI/CD：GitHub Actions
- コードレビュー：CodeRabbit

## Phase 0 成果物の定義

- 古代ギリシャ語の不明語形を入力すると、候補レンマと適用ルールを返すWebアプリケーション
- アッティカ⇔ドーリス方言の音韻変化ルール50件を含むYAMLデータファイル
- 適用ルール（ā→ē、-ας→-ης）が日本語と英語で表示される
- 少なくとも5つの異なるテストケースで正しい結果が得られる
- GitHub公開リポジトリ（英語README付き）
- デモサイト（Render等にデプロイ）

## Phase 1の目標

- 辞書データの拡充（LSJから25万レンマを取り込み）
- papyri.info EpiDoc XMLからの訓練データ自動抽出パイプライン構築
- REST APIの正式公開（認証、レート制限、ドキュメント）
- Zenodoへの音韻ルールセット初版公開（DOI取得）
- papyri.info GitHubへのIssue投稿（アノテーション提案)
